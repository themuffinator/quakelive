# Launcher Awesomium migration audit

## 1. Awesomium entry point inventory

- **WebCore bootstrap** – `sub_4f2d30` first constructs an `Awesomium::WebConfig`, calls `Awesomium::WebCore::Initialize`, aborts on failure, and later shuts the core down from `sub_4f2a60`. The global `data_12d304c` stores the returned `WebCore*` for later calls, and `data_12d3560` caches the restored cursor while the web view is active.【F:references/hlil/quakelive/quakelive_steam.exe/quakelive_steam.exe_hlil_split/quakelive_steam.exe_hlil_part05.txt†L15882-L15936】【F:references/hlil/quakelive/quakelive_steam.exe/quakelive_steam.exe_hlil_split/quakelive_steam.exe_hlil_part05.txt†L15747-L15757】
- **Custom `WebSession` preferences** – once the core exists, the launcher resolves `fs_homepath`, allocates a `WebPreferences` block, and uses the `WebCore` vtable entry `(*(*data_12d304c)...)` to create a session rooted under that path. The returned session pointer is cached as `data_12d3044`, immediately initialized, and receives bespoke data sources before any view is created.【F:references/hlil/quakelive/quakelive_steam.exe/quakelive_steam.exe_hlil_split/quakelive_steam.exe_hlil_part05.txt†L15945-L15993】
- **Launcher data sources** – two custom `Awesomium::DataSource` implementations are registered on the session: a `DataPakSource` backed by `web.pak`, and a Steam-backed source created by `sub_464300`. Both are inserted with `WebSession::AddDataSource` (`(*session->vtable+0x10)`), and a global `QLResourceInterceptor` instance is plugged into `WebCore` to intercept anything that the registered sources cannot satisfy.【F:references/hlil/quakelive/quakelive_steam.exe/quakelive_steam.exe_hlil_split/quakelive_steam.exe_hlil_part05.txt†L15969-L16005】【F:references/hlil/quakelive/quakelive_steam.exe/quakelive_steam.exe_hlil_split/quakelive_steam.exe_hlil_part02.txt†L36769-L36835】
- **View creation and helpers** – the launcher instantiates a single `WebView` via `(*(*data_12d304c + 4))(width, height, session, 0)` using globals `data_1743bcc/bd0` for the viewport size. It then installs an `Awesomium::JSMethodHandler` (`QLJSHandler`) alongside dialog, view, and load listener shims. A polling loop (`sub_431a10` + `Sleep(100ms)`) waits for the view to finish priming, emitting a fatal error if initialisation takes longer than ten attempts.【F:references/hlil/quakelive/quakelive_steam.exe/quakelive_steam.exe_hlil_split/quakelive_steam.exe_hlil_part05.txt†L16001-L16040】
- **Frame pump** – rendering flows through `sub_4f2b40`, which synchronises the Awesomium bitmap surface with the renderer and pushes the pixel data into the engine texture when the Awesomium surface reports new content. Cursor updates and mouse capture state are mirrored into engine globals to keep the launcher and in-game UI consistent.【F:references/hlil/quakelive/quakelive_steam.exe/quakelive_steam.exe_hlil_split/quakelive_steam.exe_hlil_part05.txt†L15804-L15839】

## 2. JavaScript ↔ native bindings

- **Handler registration** – `QLJSHandler` exposes the `Awesomium::JSMethodHandler` vtable and is registered with `WebView::set_js_method_handler` (vtable slot `0x12c`). This handler acts as the conduit between the DOM and the launcher’s native systems (`data_12d2670`).【F:references/hlil/quakelive/quakelive_steam.exe/quakelive_steam.exe_hlil_split/quakelive_steam.exe_hlil_part05.txt†L16003-L16012】【F:references/hlil/quakelive/quakelive_steam.exe/quakelive_steam.exe_hlil_split/quakelive_steam.exe_hlil_part05.txt†L15422-L15429】
- **Window object caching** – `sub_4f23e0` resolves `document.location` on demand, and stores the resulting window object in `data_12d3078`, allowing later calls to update `window.location.hash` from native code without re-querying the DOM. This is also the staging point for outbound event publication via the `EnginePublish` method the launcher injects.【F:references/hlil/quakelive/quakelive_steam.exe/quakelive_steam.exe_hlil_split/quakelive_steam.exe_hlil_part05.txt†L15433-L15470】【F:references/hlil/quakelive/quakelive_steam.exe/quakelive_steam.exe_hlil_split/quakelive_steam.exe_hlil_part05.txt†L16173-L16179】
- **Outbound events** – `sub_4f3260` assembles an `Awesomium::JSArray` payload (notification name, optional parameters, and an optional JSON structure) then invokes `window.EnginePublish` to deliver notifications to the UI. Errors from missing window/view objects are logged via the `PublishEvent failed` strings compiled into the binary.【F:references/hlil/quakelive/quakelive_steam.exe/quakelive_steam.exe_hlil_split/quakelive_steam.exe_hlil_part05.txt†L16080-L16140】
- **Inbound method table** – the JS method handler delegates into a native interface table at `data_12d2670`. Each exported method is a thin shim that validates the pointer and dispatches to a fixed vtable slot:
  - Lifecycle (`sub_4f1ee0`, `sub_4f1ef0`) – focus/blur entry points used by the launcher to pause/resume rendering.【F:references/hlil/quakelive/quakelive_steam.exe/quakelive_steam.exe_hlil_split/quakelive_steam.exe_hlil_part05.txt†L15092-L15106】
  - Input (`sub_4f1f10`, `sub_4f1f30`, `sub_4f1f50`, `sub_4f1f70`) – propagate keyboard and mouse data, with positional arguments passed through to the engine cursor subsystem.【F:references/hlil/quakelive/quakelive_steam.exe/quakelive_steam.exe_hlil_split/quakelive_steam.exe_hlil_part05.txt†L15109-L15148】
  - Navigation and text IO (`sub_4f2040`, `sub_4f20e0`, `sub_4f2120`, `sub_4f21e0`, `sub_4f2220`, `sub_4f2280`) – pull or store strings and JSON blobs for authentication, store browsing history, and service clipboard-like APIs by copying into caller-provided buffers.【F:references/hlil/quakelive/quakelive_steam.exe/quakelive_steam.exe_hlil_split/quakelive_steam.exe_hlil_part05.txt†L15194-L15354】
  - View mode toggles (`sub_4f20a0`, `sub_4f20c0`, `sub_4f22c0`) – enable or disable specialised render states and cursor capture without directly touching Awesomium’s API.【F:references/hlil/quakelive/quakelive_steam.exe/quakelive_steam.exe_hlil_split/quakelive_steam.exe_hlil_part05.txt†L15220-L15366】
- The method table contract implies a fixed structure describing the launcher bridge (pointer, vtable, state flags). Any in-engine replacement must provide the same slots so that scripts expecting the legacy bridge continue to function.

## 3. Resource loading map

| Awesomium fetch path | Legacy behaviour | GPL engine analogue |
| --- | --- | --- |
| `web.pak` via `DataPakSource` | Opens the `web.pak` archive under the homepath and serves static HTML/JS assets out of Awesomium’s virtual file system.【F:references/hlil/quakelive/quakelive_steam.exe/quakelive_steam.exe_hlil_split/quakelive_steam.exe_hlil_part05.txt†L15969-L16000】 | Use `FS_FOpenFileRead` / `FS_Read` to stream from PK3/PAK archives in `src/code/qcommon/files.c`, mirroring the search path logic already present in the GPL UI VM.【F:src/code/qcommon/files.c†L988-L1160】 |
| `steam://` source (`SteamDataSource`) | Registers a Steam callback (`SteamAPI_RegisterCallback`) and satisfies avatar/image requests by proxying through Steam Friends/HTTP APIs, caching responses in allocator-backed buffers.【F:references/hlil/quakelive/quakelive_steam.exe/quakelive_steam.exe_hlil_split/quakelive_steam.exe_hlil_part02.txt†L36769-L36835】 | Implement a VM-side loader that queries Steam via `trap_Steam_RequestURL` (or an equivalent new syscall) and writes into the same menu image cache used by `UI_DrawHandlePic` / `trap_R_RegisterShaderNoMip`. |
| Generic HTTP fallback (`QLResourceInterceptor`) | Intercepts unresolved requests, rewrites them against `fs_webpath`, and ultimately uses `Awesomium::ResourceResponse::Create` on top of regular filesystem reads. Screenshot requests are rewritten into `fs_homepath/screenshots/...` paths before reading bytes off disk.【F:references/hlil/quakelive/quakelive_steam.exe/quakelive_steam.exe_hlil_split/quakelive_steam.exe_hlil_part01.txt†L46762-L47013】 | Mirror the logic with `FS_FOpenFileRead` and `trap_FS_Read` for HTTP-equivalent assets; screenshot access maps naturally to the renderer’s screenshot directory already exposed to the VM through `trap_FS_FOpenFile`. |

## 4. In-engine menu prototype & validation checklist

**Prototype outline**

1. **Entry point** – add a new menu definition (e.g. `ui/launcher.menu`) and expose it through `_UI_SetActiveMenu` in `ui_main.c`, using the existing VM dispatcher in `vmMain` to open the menu when the game starts in launcher mode.【F:src/code/ui/ui_main.c†L168-L199】
2. **Layout & rendering** – build the menu with standard `itemDef_t` controls and reuse the renderer hooks (`UI_DrawHandlePic`, `Text_Paint`) to display Awesomium-derived textures, authenticated user details, and navigation buttons.【F:src/code/ui/ui_atoms.c†L415-L451】【F:src/code/ui/ui_main.c†L348-L406】
3. **Input plumbing** – forward `_UI_KeyEvent` and `_UI_MouseEvent` into the prototype menu so it can toggle focus and capture mouse motion while mirroring the launcher’s cursor-lock semantics.【F:src/code/ui/ui_main.c†L5175-L5220】
4. **State bridge** – implement a lightweight C struct mirroring the launcher bridge (matching the `data_12d2670` vtable layout) inside the UI VM so script calls keep working. Where Awesomium previously pushed DOM events, call `Menus_OpenByName` / `Menus_CloseByName` and update cvars instead.【F:references/hlil/quakelive/quakelive_steam.exe/quakelive_steam.exe_hlil_split/quakelive_steam.exe_hlil_part05.txt†L15092-L15366】【F:src/code/ui/ui_shared.c†L1000-L1033】

**Validation criteria**

- **Rendering** – confirm parity with the legacy launcher by validating that every menu state renders within the 640×480 virtual canvas and scales via `UI_AdjustFrom640`. Use automated captures comparing `UI_DrawHandlePic` output against baseline textures.【F:src/code/ui/ui_atoms.c†L408-L451】
- **Input** – verify keyboard/mouse focus handling by exercising `_UI_KeyEvent` and `_UI_MouseEvent`, ensuring escape handling and cursor confinement behave identically to the Awesomium window.【F:src/code/ui/ui_main.c†L5175-L5220】
- **Localisation** – reuse `Text_Paint` and `Text_PaintCenter_AutoWrapped` so translated strings stay within bounds and fonts respect glyph metrics. Regression tests should cover English plus any wide-character locales the original launcher targeted.【F:src/code/ui/ui_main.c†L348-L406】【F:src/code/ui/ui_main.c†L5345-L5387】
- **Auth/navigation flows** – script harness should call into the recreated bridge table (matching the offsets used in `sub_4f1f10`–`sub_4f2280`) and assert the same responses the launcher delivered (success/failure codes, buffer writes, view toggles).【F:references/hlil/quakelive/quakelive_steam.exe/quakelive_steam.exe_hlil_split/quakelive_steam.exe_hlil_part05.txt†L15109-L15354】

These notes capture the work necessary to replicate the launcher’s Awesomium surface inside the GPL engine, while documenting the contracts that existing JavaScript expects and the engine subsystems that must replace Awesomium’s data loading behaviour.
