## Alignment and gaps relative to `src/`
- **Shared gameplay architecture:** `qagame` HLIL and native DLLs map cleanly to `src/code/game/`, where Quake III still defines `g_main`, weapon logic, and bot AI under the VM API. This provides a one-to-one reference for porting gameplay changes from Quake Live into the GPL code or vice versa.【F:references/hlil/quakelive/qagamex86.dll/qagamex86.dll.bndb_hlil.txt†L1-L4】【F:src/code/game/g_main.c†L41-L68】
- **Client game parity:** The `cgame` dump aligns with `src/code/cgame/`’s `vmMain` dispatcher, reinforcing that most rendering and prediction hooks retain the same entry points even though Quake Live executes natively.【F:references/hlil/quakelive/cgamex86.dll/cgamex86.dll_hlil.txt†L1-L4】【F:src/code/cgame/cg_main.c†L41-L69】
- **UI layering:** Quake Live’s `uix86.dll` corresponds to both the C UI code and the script-driven menu files still present in `src/`. This demonstrates that UI logic spans compiled code (`ui_main.c`, `ui_shared.c`) and data (`menus.txt`).【F:references/hlil/quakelive/uix86.all/uix86.dll_hlil.txt†L1-L4】【F:src/code/ui/ui_main.c†L8-L36】【F:src/ui/menus.txt†L1-L18】
- **Engine divergence:** `quakelive_steam.exe` has no direct analogue in `src/`; the GPL release retains the classic VM loader and lacks Steam, Chromium, or media codec integrations. Any attempt to reproduce Quake Live features must either reimplement these services or hook into new platform layers absent from `src/code/client/` and `src/code/server/`.【F:references/hlil/quakelive/quakelive_steam.exe/quakelive_steam.exe_hlil.txt†L1-L4】【F:src/code/client/cl_main.c†L41-L68】【F:src/code/server/sv_init.c†L40-L74】
- **Platform services gap:** Support binaries such as `awesomium.dll`, `steam_api.dll`, and the FFmpeg libraries provide functionality outside the scope of the Quake III GPL release, highlighting areas that require reverse-engineered shims or substitutes if those behaviours are desired.【9d6ea8†L12-L24】
- **Configuration data differences:** Quake Live’s generated configs (`syncerror.cfg`, named player configs) expose additional bindings absent from the vanilla Quake III `baseq3`, indicating live-service defaults that may need manual reproduction when working from the GPL sources.【9d6ea8†L4-L8】【F:assets/quakelive/baseq3/syncerror.cfg†L1-L19】

## Race checkpoints
### Authoring `race_point`
Race mode maps drop `race_point` entities anywhere a checkpoint should exist; during spawn the server snaps the trigger to the ground, applies a fixed 48-unit tall bounding box, and registers the point so that players can touch it in-order.【F:src/code/game/g_race.c†L57-L81】【F:src/code/game/g_race.c†L582-L601】 Each checkpoint caches its origin, `target`, and `targetname` strings inside `racePointInfo_t`, allowing administrators to tag a point that leads to a mover or signal a finishing volume.【F:src/code/game/g_race.c†L90-L108】【F:src/code/game/g_local.h†L607-L613】 The level state preserves a deterministic array of pointers plus the metadata cache (`level.racePoints[]`, `level.racePointInfo[]`, `level.racePointCount`), so map authors can rely on spawn order to define split ordering for the HUD.【F:src/code/game/g_local.h†L735-L738】

### Admin and console commands
During play, the `racepoint` console command (client-side) routes through `G_RaceAdminCommand` to spawn checkpoints at the player’s feet, dump the cached metadata (including target strings), or clear the entire array. All of those flows rebroadcast metadata via `admin_race_point_%i` commands followed by `race_init` so that every client sees the new layout.【F:src/code/game/g_race.c†L640-L739】 Dedicated servers expose the same verbs plus server-driven `race_init`, `race_info`, and `admin_race_point_%i` handlers, allowing remote administrators to dump or clear checkpoints without logging in as a player.【F:src/code/game/g_cmds.c†L2444-L2505】【F:src/code/game/g_svcmds.c†L498-L543】 When a dump or clear runs, the server prints each checkpoint’s index, position, target, and targetname so mappers can cross-reference brushes inside Radiant.【F:src/code/game/g_race.c†L610-L650】

Administrators can also refresh the match factory registry without a `map_restart` by issuing the `reload_factories` console command. The handler re-parses `scripts/factories.txt` plus all supplemental `.factories` files, reapplies the previously selected factory (respecting the existing “do not clear active factory” guard), and logs the same diagnostics used during startup so operators can see which presets loaded successfully.【F:src/code/game/g_factory.c†L820-L887】【F:src/code/game/g_factory.c†L933-L985】【F:src/code/game/g_svcmds.c†L480-L486】 This mirrors the Quake Live DLL workflow that let CI or remote admins adjust presets mid-session without recycling the VM.

### Metadata replication and HUD consumption
Every time the checkpoint list changes, `G_RaceBroadcastInitCommand` sends `race_init` followed by per-point `admin_race_point_%i` payloads that include the stored origin/target strings, while `G_RaceSendInfoCommand` refreshes the `race_info` payload that carries total checkpoint counts and leader splits.【F:src/code/game/g_race.c†L117-L175】 Client game code resets its local cache when `race_init` arrives, updates split data when `race_info` arrives, and records each `admin_race_point_%i` payload so HUD code can render checkpoint indicators in the correct order.【F:src/code/cgame/cg_servercmds.c†L67-L85】【F:src/code/cgame/cg_servercmds.c†L1377-L1389】 This mirrors the behaviour described in `docs/race_admin.md`, giving mappers a single workflow for spawning checkpoints, dumping their metadata, and verifying that the HUD reflects the same points.【F:docs/race_admin.md†L1-L28】
